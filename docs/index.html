The problem it solves

Without a tool to read your code, AI assistants guess. yoyo stops the guessing.

Without yoyo

  • Wrong file pathsAI assistants invent plausible-sounding paths that don't exist in your project.
  • Stale function namesAnswers come from training data, not your current code on disk.
  • 20-minute onboardingTo understand a 400-file project, the AI reads dozens of files one by one.
  • Missed call chainsWithout a call graph, the AI can't tell you what breaks when you rename a function.

With yoyo

  • Exact file paths and line numbersEvery answer comes from the AST. File, line, and byte offset are exact.
  • Current function namesyoyo reads your files on disk. The index is always in sync after every edit.
  • 30-second onboarding8 parallel tool calls replace 15–20 minutes of directory browsing and grepping.
  • Structural impact in one stepimpact shows who calls yours, what breaks, and how a request path lands.

What it does

yoyo runs one command to parse your project, then serves the results over MCP.

Parses your code with ast-grep

Builds an index of every function, its file, line range, byte offsets, complexity score, and outgoing calls. Supports 15 languages: TypeScript, JavaScript, Rust, Python, Go, C, C++, C#, Java, Kotlin, PHP, Ruby, Swift, Bash, and Zig.

Gives Claude exact answers

inspect returns the exact symbol, file outline, or line range you need. search finds call sites and identifiers across hundreds of files. No guessing.

Traces call chains

impact finds everything that touches a symbol or traces an endpoint into its downstream call chain (Rust + Go today for deep endpoint chains).

Vertical slice in one call

impact(endpoint=...) resolves an HTTP endpoint to its handler, then traces the call chain to db/http/queue boundaries — replacing several sequential tool calls with one. Handler source optional.

Edits code safely

change routes edits, renames, moves, deletes, creates, and adds through one task-shaped write verb. Reindexes automatically so the next read is always fresh.

Audits dead code

health reports functions that are never called, functions with high complexity and high fan-out, and functions with duplicate names across files. Public API is excluded — only genuinely unreachable code is flagged.

Semantic search — no API key

ask finds functions by intent: "semaphore acquisition", "spawn blocking task", "async runtime builder". Powered by local ONNX embeddings (fastembed AllMiniLML6V2). No network, no key.

Runs entirely on your machine

Single static binary. No network requests, no API keys, no telemetry. The index is a plain JSON file in your project root.

How Claude works with yoyo

Claude treats yoyo as its primary interface to the codebase. You don't manage the tool calls.

1. Bootstrap

Claude calls boot and index on first contact — lean catalog, task routing, and concurrency rules up front. Calls deeper guidance only on demand. No file reading, no grepping.

2. Read

inspect, search, and ask replace grep and ad hoc file reads. Structured data from the AST index — not line matches from raw source.

3. Understand

impact, routes, and health answer structural questions no text tool can: who calls this? what does this touch? is this dead?

4. Write

change mutates code and auto-reindexes. One write verb, many actions. No direct file edits.

Result: Claude answers from facts, not memory. No hallucinated file paths. No stale function names.

🪀 Philosophy

One tool is fine. Combinations are deadly.

In yoyo tournaments, a yoyo is just a spinning disk on a string. The magic is in the combinations — string wraps, body movements, timing layered together. A single trick is fine. Fifty moves chained in sequence is something else entirely.

yoyo works the same way. Each tool does one thing cleanly. The power is in how your agent orchestrates them:

CombinationWhat it does
search → inspect → changeFind it, read it, change it ⚡
impact → health → changeWho calls this? Is it dead? Remove it safely.
impact → changeTrace the full request path, fix it end-to-end in one shot.
index → ask → mapWhere does this new function belong?
map → routes → changeUnderstand the shape, find the gap, fill it.

No single tool is the point. The orchestration is.

  | tar xz
sudo mv yoyo-x86_64-unknown-linux-gnu /usr/local/bin/yoyo

Build from source

git clone https://github.com/avirajkhare00/yoyo.git
cd yoyo && cargo build --release
sudo cp target/release/yoyo /usr/local/bin/yoyo

Quick start

# index your project
yoyo bake --path /path/to/your/project

# find a function by name
yoyo inspect --path /path/to/your/project --name myFunction

# search across all files
yoyo search --path /path/to/your/project --query myFunction

Blog

Design decisions, trade-offs, and things learned building yoyo.

March 9, 2026 • Release • v1.3.8

The compiler guard: patch now proves it compiles

Before v1.3.8, patch had a pre-write AST check. It would parse the new content with tree-sitter before touching the file — catching syntax errors before they landed on disk. That was good. It was not enough.

Syntax is the easy case. The harder failures — wrong identifiers, mismatched signatures, undefined types, moved imports — all pass the AST check and only surface when the compiler runs. In a session where an AI agent patches ten functions, one type error in the middle means the project no longer builds. The agent has to backtrack, find the bad patch, fix it, and rebuild — often losing context along the way.

v1.3.8 adds a compiler guard. The sequence is now: write → compile → roll back on error. If the compiler rejects the change, the file is restored to its original content and patch returns an error. A successful patch call is a guarantee: the file compiles. 

// what happens inside patch now
1. write new content to file
2. run compiler (cargo check / go build / zig ast-check)
3a. compiler passes → reindex, return success
3b. compiler fails → restore original file, return error

Languages covered: Rust (cargo check), Go (go build), Zig (zig ast-check). For other languages, the AST pre-check still runs — syntax errors are caught, type errors are not. Extending the guard to more languages means finding a fast, zero-config compile check for each one.

The design principle here is the same as graph_delete blocking on live callers: make the unsafe case structurally impossible, not just documented. An agent that patches Rust code cannot leave the project in a broken state — not because of a rule, but because patch won't let the bad write stand.

March 10, 2026 • Design

Why script replaced pipeline

pipeline was a tool that accepted a JSON spec — an ordered list of steps, each naming a yoyo tool and its arguments — and executed them sequentially. Steps could reference earlier results via {{step_id.field}} template refs. Conditions (if) could skip steps when false. The intent was sound: encode multi-step workflows as data rather than prose, so agents could run them atomically without back-and-forth.

In practice, pipeline had a ceiling. It could sequence. It could skip. It could not loop, filter, accumulate, or branch on anything more complex than a single boolean condition. The moment a workflow needed "for each dead function, check its callers, then delete if zero" — pipeline couldn't express it. You needed a loop. Pipeline had no loop.

The patch: add loop syntax to the spec. The counter-argument: we're now designing a programming language inside JSON. Every new construct (loop, map, filter, early return) requires a new spec key, a new parser branch, and a new set of edge cases. JSON is a fine data format. It is a terrible programming language substrate.

script takes a different approach. Instead of a JSON spec, it accepts a Rhai script — a small, fast scripting language embedded directly in Rust. Rhai gives you real loops, real conditionals, closures, map and filter, early returns. The same task-shaped MCP tools (inspect, search, ask, map, routes, impact, health, change, plus boot, index, and help) are exposed as plain functions. The last expression in the script is the result. 

// find all dead functions, return the ones with zero callers
let dead = health().dead_code;
dead.filter(|f| {
    let r = impact(#{symbol: f.name});
    r.callers.len() == 0
})

This replaces a multi-turn conversation. No back-and-forth: one tool call, one structured result.

Scripts no longer expose hidden mechanism tools. They compose the same task-shaped surface the model sees everywhere else. That keeps MCP recall stable over long sessions instead of giving scripts a separate internal dialect.

The important distinction is not "script vs no script". It is task verbs vs mechanism leakage. A script should still say impact, inspect, and change, because that is the product surface the agent is supposed to remember.

That still leaves room for orchestration. A script can iterate over dead code candidates, confirm zero callers with impact, and then call change(#{action: "delete", ...}). The safety property stays structural because the underlying delete path still blocks when callers exist.

The reason pipeline was kept for as long as it was: the idea of executable specs as project memory is genuinely good. A JSON pipeline is inspectable, serialisable, and diffable in a way a script is not. The right answer may eventually be a spec format that compiles to Rhai — declarative at the surface, expressive underneath. For now, script is strictly more useful and pipeline is gone.

Net effect: fewer top-level MCP tools, more task completion per tool, and no special hidden mechanism dialect inside scripts. Fewer is better when each tool pulls more weight.
